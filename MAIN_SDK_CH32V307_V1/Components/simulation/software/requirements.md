# 控制仿真上位机需求规格说明书

## 1. 文档概述

### 1.1 文档目的

本文档用于定义“控制仿真上位机”软件的业务目标、系统范围、功能需求、接口约束、非功能要求与验收标准，为后续开发、测试、联调和维护提供统一依据。

### 1.2 适用对象

本文档适用于以下角色：

- 嵌入式控制算法开发人员
- 上位机软件开发人员
- 联调测试人员
- 项目维护人员

### 1.3 名词说明

- 下位机：运行控制算法的开发板或目标控制器
- 上位机：本项目中的桌面仿真与联调软件
- 遥测：下位机发送到上位机的状态、姿态、控制量等信息
- 反馈：上位机仿真得到并回传给下位机的被控对象状态
- 闭环联调：下位机输出控制量，上位机仿真对象响应并返回反馈量，形成完整控制回路

## 2. 项目背景

项目目标是构建一套用于控制算法验证与联调的上位机工具。在没有完整实物执行机构或复杂现场环境的情况下，通过“下位机实机运行控制算法 + 上位机实时仿真被控对象”的方式，完成控制闭环验证、参数调试、状态观测和数据记录。

当前通信链路以蓝牙串口透传或普通 UART 串口为主。下位机可采集传感器数据并运行控制律，上位机负责对象动力学仿真、图形化展示、命令交互与数据归档。

## 3. 建设目标

系统应满足以下目标：

- 支持下位机与上位机之间稳定的双向通信
- 支持至少一种基础被控对象模型的实时仿真
- 支持控制算法切换、参考值设置和运行控制
- 支持对关键变量进行实时图形化显示
- 支持记录联调过程数据并导出为 CSV
- 支持后续扩展更多模型、更多协议字段和更多可视化能力

## 4. 系统范围

### 4.1 范围内

- 上位机图形界面
- 串口连接与通信管理
- 下位机遥测接收与解析
- 上位机被控对象仿真
- 仿真反馈回传
- 控制命令输入与下发
- 实时波形显示
- 3D 模型显示
- 运行状态显示
- CSV 数据记录

### 4.2 当前阶段不在范围内

- 云端平台接入
- 多机联网协同
- 复杂三维可视化场景
- 完整离线回放系统
- 与实际动力执行机构直接联动的工业级控制系统

## 5. 系统角色与职责

### 5.1 下位机职责

- 采集姿态或其他传感器数据
- 周期执行控制算法
- 产生控制输出 `u_cmd`
- 上报遥测信息
- 接收上位机返回的仿真反馈数据
- 响应来自上位机的控制命令

### 5.2 上位机职责

- 提供串口连接与会话管理
- 解析下位机遥测数据
- 执行被控对象仿真
- 回传仿真反馈数据
- 提供控制算法选择和命令下发能力
- 提供运行状态展示和波形可视化
- 提供联调数据记录功能

## 6. 业务场景

### 6.1 算法联调场景

开发人员通过上位机连接下位机，选择控制算法，设置参考值，启动控制过程，观察控制输出与仿真反馈之间的动态响应，从而验证算法的基本稳定性和跟踪性能。

### 6.2 协议联调场景

开发人员使用上位机验证串口协议、字段定义、时序行为和应答机制，确认上下位机之间的数据帧可稳定互通。

### 6.3 数据采集场景

测试人员在控制运行过程中记录遥测数据与仿真数据，用于离线分析、参数整定和问题复现。

## 7. 功能需求

### 7.1 串口连接管理

系统应支持串口连接管理功能，包括：

- 扫描当前系统可用串口
- 选择串口号
- 选择波特率
- 打开串口连接
- 主动断开串口连接
- 显示当前连接状态

系统应支持蓝牙串口透传和普通串口两类链路，均按 UART 连接方式处理。

### 7.2 通信模式配置

系统应支持仿真反馈的两种发送模式：

- 二进制帧模式
- 文本行模式

用户应可在界面上切换反馈发送模式。默认模式可配置，当前建议默认启用二进制发送。

### 7.3 控制命令下发

系统应支持通过图形界面和命令输入框向下位机发送控制命令。

首版应至少支持以下命令：

- `ALG PID`
- `ALG LADRC`
- `ALG OPEN_LOOP`
- `SET REF <val>`
- `RUN 1`
- `RUN 0`
- `GET STATUS`

系统应允许用户手工输入其他原始命令，例如：

- `SET KP <val>`
- `SET KI <val>`
- `SET KD <val>`
- `SAVE`

命令发送后，界面应能显示已发送内容和下位机返回信息。

### 7.4 算法选择

系统应提供算法选择控件，首版至少支持以下算法类型：

- `PID`
- `LADRC`
- `OPEN_LOOP`

界面可使用中文展示名称，但发送给下位机的协议值必须保持与固件约定一致。

### 7.5 参考值设置

系统应支持用户输入参考值并发送至下位机。

参考值控件应支持：

- 数值输入
- 小数设置
- 基本范围限制

参考值发送后，应在控制台中保留对应命令记录。

### 7.6 运行控制

系统应支持控制过程的启动与停止。

启动时应向下位机发送运行命令，停止时应向下位机发送停止命令。运行状态应在界面状态区实时显示。

### 7.7 遥测接收与解析

系统应支持接收下位机发送的遥测数据，并能够同时解析以下格式：

- 二进制帧
- `key=value` 形式文本行
- JSON 文本行
- 普通应答文本行

应至少支持以下遥测字段：

- `timestamp`
- `roll`
- `pitch`
- `yaw`
- `u_cmd`
- `ref`
- `feedback`
- `algo_id`
- `run_state`

对于未知但可解析为数值的字段，系统应保留其扩展能力，便于后续添加新通道。

### 7.8 仿真对象计算

系统应在上位机内部实现至少一个被控对象模型。首版应至少具备：

- 单自由度深度或高度模型
- 控制输入驱动
- 阻尼项
- 偏置项
- 扰动项
- 测量噪声

仿真输出应至少包含：

- `depth`
- `depth_rate`
- `disturbance`

### 7.9 仿真反馈回传

系统应按固定周期将仿真反馈发送给下位机。

首版建议发送频率为 20 Hz 至 50 Hz。当前默认设计值可取 25 Hz。

反馈数据应支持：

- 二进制编码发送
- 文本编码发送

### 7.10 实时波形显示

系统应提供多通道实时波形显示能力。

波形功能至少应包括：

- 多通道曲线显示
- 通道开关
- 图例显示
- 时间窗口调整
- 暂停显示
- 清空显示

首版建议同时支持不少于 8 个通道显示。建议默认通道包括：

- 参考值
- 反馈值
- 控制输出
- 滚转角
- 俯仰角
- 偏航角
- 深度变化率
- 扰动

### 7.11 3D 模型显示

系统应支持三维模型可视化能力，用于更直观地展示姿态变化。

首版应至少满足：

- 显示一个简化三维机体模型
- 根据 `roll`、`pitch`、`yaw` 更新模型姿态
- 显示当前姿态角数值
- 可与波形界面同时显示

三维模型显示主要服务于联调观察，不作为控制闭环计算输入。

### 7.12 状态监视

系统应提供运行状态面板，至少显示以下信息：

- 当前算法
- 当前运行状态
- 参考值
- 反馈值
- 控制输出
- 接收帧数
- 发送帧数
- 解析错误数
- 通信延迟
- 通信超时状态

### 7.13 数据记录

系统应支持将联调过程中的遥测数据和仿真数据保存到 CSV 文件。

记录功能应满足：

- 支持开始记录
- 支持停止记录
- 允许用户选择保存路径
- 首次写入时自动生成表头
- 运行过程中自动刷新落盘，降低异常退出时的数据丢失风险

### 7.14 日志与错误提示

系统应在控制台区域显示以下类型的信息：

- 已发送命令
- 串口连接状态
- 下位机普通输出
- 协议应答
- 解析错误或串口错误

错误信息应尽量对使用者可理解，便于快速定位问题。

## 8. 接口与协议需求

### 8.1 物理链路

通信链路采用串口方式，支持：

- USB 转串口
- 蓝牙串口透传

### 8.2 下位机到上位机协议

系统应支持接收以下两类数据：

1. 二进制帧
2. 文本行

二进制帧建议结构如下：

```text
AA 55 | msg_id | len | seq | payload | crc16
```

其中建议定义：

- `msg_id = 0x01`：遥测数据
- `msg_id = 0x20`：文本应答

当前首版遥测载荷可按以下顺序定义：

- `timestamp_ms`
- `roll`
- `pitch`
- `yaw`
- `u_cmd`
- `ref`
- `algo_id`
- `run_state`

文本行应支持：

- `key=value,key=value`
- JSON 行文本
- `OK`
- `ERR <code> <msg>`

### 8.3 上位机到下位机协议

上位机应支持向下位机发送仿真反馈数据。

反馈消息类型建议定义为：

- `msg_id = 0x10`

反馈内容至少包括：

- `timestamp`
- `depth`
- `depth_rate`
- `disturbance`

### 8.4 校验机制

二进制帧应支持校验机制。首版建议使用 `CRC16-CCITT`。

### 8.5 时序要求

控制算法执行与通信收发应相互解耦。

建议时序如下：

- 下位机控制周期：约 100 Hz
- 上位机 UI 刷新周期：约 30 Hz
- 上位机反馈发送周期：约 25 Hz
- 通信超时判定：约 300 ms

## 9. 安全性与鲁棒性要求

系统应具备基本的联调安全保护能力。

### 9.1 通信超时

若在超时时间内未收到新的有效遥测，系统应在界面中标记通信超时状态。

下位机侧建议在协议层配合以下策略之一：

- 控制量清零
- 控制量限幅保持
- 进入保护状态

### 9.2 非法数据处理

对于以下情况，系统不应异常崩溃：

- 文本格式错误
- JSON 格式错误
- CRC 校验失败
- 字段缺失
- 字段类型不合法

系统应累计解析错误次数，并继续处理后续数据。

### 9.3 运行中参数变更

对于算法切换和关键参数更新，系统应支持在联调流程中进行控制，但建议下位机根据自身安全策略决定是否允许在运行状态下立即生效。

## 10. 非功能需求

### 10.1 实时性

- 波形显示在正常联调负载下不应出现明显卡顿
- 界面操作应具有可接受的响应速度

### 10.2 稳定性

- 系统应支持连续运行至少 10 分钟不崩溃
- 串口反复连接与断开后，应仍能继续使用

### 10.3 可维护性

- 协议字段应具备扩展空间
- 通道定义应易于扩展
- 被控对象模型应可替换
- UI 面板应具备模块化组织方式

### 10.4 可移植性

- 首版以 Windows 平台为主
- Python 运行环境应便于部署和复现

### 10.5 可观测性

- 应能观察收发帧数
- 应能观察延迟和超时状态
- 应能查看串口和协议错误信息

## 11. 验收标准

### 11.1 启动验收

- 在安装依赖后，执行 `python run.py` 可正常打开主界面
- 主界面包含串口区、控制台区、状态区和波形区

### 11.2 通信验收

- 可成功扫描并连接目标串口
- 下位机持续发送数据时，上位机能稳定接收并更新状态
- 双向通信连续运行 10 分钟无明显协议失步

### 11.3 控制验收

- 可正确发送算法切换命令
- 可正确发送参考值命令
- 可正确发送启动和停止命令
- `GET STATUS` 的结果与界面状态保持一致

### 11.4 显示验收

- 至少 8 个通道可同时显示
- 支持暂停、清空和时间窗口调整
- 运行状态、通信延迟、解析错误数可见
- 3D 模型可随姿态数据正确联动

### 11.5 记录验收

- 可正常开始和停止 CSV 记录
- 生成的 CSV 文件可被常见工具读取
- 记录内容至少覆盖遥测核心字段和仿真输出字段

### 11.6 容错验收

- 串口断开后程序不崩溃
- 接收非法帧后程序不崩溃
- 超时情况下界面可给出明确提示

## 12. 当前版本实现映射

以下内容为当前版本已实现或部分实现的能力说明：

- 已实现：
  - 串口连接管理
  - 二进制 / 文本两种反馈发送模式
  - 基础控制命令发送
  - 单自由度深度对象仿真
  - 实时波形显示
  - 3D 姿态模型显示
  - 状态面板
  - CSV 数据记录
  - 二进制与文本遥测解析
- 部分实现：
  - 参数配置仍主要依赖手工命令输入
  - 回放功能尚未形成完整产品能力
  - 事件标记与高级分析功能尚未完善

## 13. 后续扩展建议

- 增加参数配置专用界面
- 增加多对象模型切换能力
- 增加离线回放功能
- 增加事件标记与注释能力
- 增加图表缩放、游标和数据导出能力
- 增加更多协议字段及自定义通道映射
- 增加配置文件与用户偏好保存功能
